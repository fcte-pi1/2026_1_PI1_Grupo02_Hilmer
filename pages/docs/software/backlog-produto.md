## Backlog do Produto

O *Backlog* do Produto consolida as necessidades e funcionalidades do sistema Micromouse, sendo estruturado para garantir a rastreabilidade e a priorização adequada das entregas. As funcionalidades do sistema foram documentadas diretamente utilizando-se a técnica de Histórias de Usuário (HU), permitindo a compreensão clara das necessidades dos atores sob a perspectiva de valor de negócio.

Para a priorização dos itens, adotou-se o método MoSCoW, classificando as funcionalidades em *Must Have* (essenciais), *Should Have* (importantes, mas não vitais) e *Could Have* (desejáveis). Adicionalmente, os requisitos não-funcionais (RNF) foram definidos sob critérios objetivos, evitando descrições subjetivas e focando em métricas mensuráveis de desempenho, como tempo de resposta e estabilidade da conexão.

Todas as Histórias de Usuário (HUs) possuem descrição estruturada no formato **Eu–Como–Para**, critérios de aceitação e protótipos de interface devidamente documentados. A estrutura completa do Backlog do Produto encontra-se disponível no [Backlog do Projeto no GitHub](https://github.com/orgs/fcte-pi1/projects/16/views/1), permitindo rastreabilidade e acompanhamento detalhado do desenvolvimento do projeto.

Vale ressaltar que este documento foca nos épicos e requisitos relacionados ao desenvolvimento de **Software**. A parte complementar do backlog, com foco em sistemas embarcados, pode ser encontrada na seção de [Embarcado](../eletronica/embarcado.md).

As tabelas a seguir apresentam a divisão do backlog organizada por épicos de desenvolvimento.

### Épico B: Coletar e Transmitir Dados de Telemetria

| ID | Título da História | Priorização (MoSCoW) | História de Usuário | Critério de Aceite |
| :---- | :---- | ----- | :---- | :---- |
| **HU-08** | Recepção de Dados | Must Have | Como avaliador, eu quero que o sistema receba os dados do Micromouse para que sejam exibidos e registrados. | **CA-08-01: Dado** que o Micro Mouse está configurado para transmitir dados via conexão sem fio; **Quando** o Micromouse inicia o envio de pacotes de telemetria durante a execução; **Então** o sistema web deve processar cada pacote e disponibilizá-lo para a interface de monitoramento sem perda de dados. |
| **HU-09** | Monitoramento | Should Have | Como avaliador, eu quero que o sistema indique visualmente o status de conexão atual (online/offline). | **CA-09-01: Dado** que o sistema web está monitorando ativamente uma conexão; **Quando** o sinal do Micromouse não é recebido por um intervalo superior a 3 segundos; **Então** o sistema web deve mudar o indicador de status para "Offline" e exibir um alerta visual de interrupção na tela. |
| **HU-10** | Validação | Must Have | Como avaliador, eu quero que o sistema valide os dados recebidos para evitar inconsistências. | **CA-10-01:** **Dado** que o sistema web está recebendo pacotes de telemetria do Micromouse; **Quando** um pacote for recebido, **então** o sistema web deve verificar se os campos obrigatórios estão presentes e em formato válido antes de disponibilizá-los para visualização ou armazenamento; **CA-10-02:** Dado que um pacote de telemetria possua dados inválidos, incompletos ou inconsistentes, Quando o sistema web realizar a validação, Então o pacote não deve ser armazenado no banco de dados nem exibido como dado válido na interface.  **CA-10-03:** **Dado** que um pacote inválido seja identificado, **Quando** a validação falhar, **Então** o sistema web deve sinalizar a falha para que a equipe consiga identificar o problema de comunicação ou formatação dos dados. |
| **HU-20** | Comunicação | Must Have | Eu como avaliador quero que o sistema web se comunique com o micromouse para obter dados. | **CA-20-01: Dado** que o micromouse e o sistema estão ligados. Quando a comunicação for iniciada; Então o sistema web e o micromouse devem estabelecer uma sessão de conexão. |

### Épico C: Exibir interface de análise da sessão da corrida

| ID | Título da História | Priorização (MoSCoW) | História de Usuário | Critério de Aceite |
| :---- | :---- | ----- | :---- | :---- |
| **HU-11** | Início de Sessão | Must Have | Como avaliador, eu quero iniciar uma sessão de monitoramento associada a uma execução do Micromouse, para que o sistema identifique a corrida em andamento. | **CA-11-01:** **Dado** que o avaliador acessa a interface web; **Quando** clica em "Iniciar Sessão" e selecione o tipo de labirinto (4x4, 8x8 ou 16x16); **Então** o sistema web deve criar uma nova sessão identificada com timestamp de início, tipo de labirinto e status "Em andamento".  **CA-11-02:** **Dado** que a sessão é encerrada (manualmente, por falha ou por conclusão do desafio); **Quando** o encerramento é confirmado; **Então** o sistema web deve registrar o timestamp de fim e alterar o status da sessão para "Concluída". |
| **HU-12** | Exibição do Trajeto | Must Have | Como avaliador, eu quero visualizar o trajeto percorrido pelo Micromouse no labirinto, para acompanhar sua navegação durante a corrida. | **CA-12-01:** **Dado** que uma sessão está ativa e dados de posição são recebidos; **Quando** o Micromouse se move para uma nova célula; **Então** o mapa do labirinto deve atualizar em tempo real, marcando as células já conhecidas e destacando a posição atual do Micromouse.  **CA-12-02:** **Dado** que o Micromouse ainda não se moveu; **Quando** a sessão é iniciada; **Então** o mapa deve exibir o labirinto com o Micromouse na posição inicial, sem células marcadas como conhecidas.  **CA-12-03:** **Dado** que a corrida é encerrada; **Quando** o avaliador acessa o histórico; **Então** o trajeto completo percorrido deve estar disponível para visualização estática no mapa. |
| **HU-13** | Indicadores de Desempenho | Must Have | Como avaliador, eu quero visualizar indicadores de desempenho em tempo real, para avaliar o comportamento do Micromouse durante a corrida. | **CA-13-01:** **Dado** que a sessão está ativa; **Quando** os dados são recebidos; **Então** devem ser exibidos: bateria (%), velocidade média (cm/s) e tempo decorrido. **CA-13-02**: **Dado** que a corrida é concluída; **Então** o tempo total deve ser exibido como tempo de conclusão fixo. |
| **HU-14** | Status do Desafio | Must Have | Como avaliador, eu quero visualizar se o desafio foi cumprido ou não, para verificar o resultado da execução em tempo real. | **CA-14-01:** **Dado** que uma sessão está ativa; **Quando** o Micromouse ainda não alcançou o objetivo; **Então** o status exibido deve ser "Desafio em andamento" ou equivalente, em destaque na tela.  **CA-14-02:** **Dado** que o sistema web recebe o sinal de que o Micromouse alcançou a área central do labirinto; **Quando** esse evento é processado; **Então** o status deve mudar imediatamente para "Desafio Cumprido " ou equivalente.  **CA-14-03:** **Dado** que a sessão é encerrada sem que o objetivo tenha sido alcançado; **Quando** o encerramento é registrado; **Então** o status final deve ser "Desafio Não Cumprido” ou equivalente. |
| **HU-15** | Alerta de Evento Crítico | Should Have | Como avaliador, eu quero receber alertas visuais imediatos quando ocorrer um evento crítico durante a corrida, para agir rapidamente. | **CA-15-01: Dado** que uma sessão está ativa; **Quando** a bateria atinge nível crítico (≤10%); **Então** um alerta visual e sonoro deve ser disparado imediatamente na tela de monitoramento. **CA-15-02: Dado** que dados de velocidade são recebidos; **Quando** a velocidade cai para zero por mais de 3 segundos sem que a corrida tenha sido encerrada; **Então** o sistema web deve exibir alerta de "Possível parada inesperada" |

### Épico D: Salvar e Consultar Dados das Corridas

| ID | Título da História | Priorização (MoSCoW) | História de Usuário | Critério de Aceite |
| :---- | :---- | ----- | :---- | :---- |
| **HU-16** | Registro de Dados Finais | Must Have | Como avaliador, eu quero que todos os dados da corrida sejam armazenados automaticamente ao final da corrida, para análise posterior. | **CA-16-01:** **Dado** que a sessão de monitoramento é encerrada; **Quando** o sistema processa o encerramento; **Então** o sistema deve salvar automaticamente todos os dados da corrida, incluindo telemetria histórica, trajeto percorrido, tempo total, status do desafio e tipo de labirinto associado **CA-16-02**: **Dado** que o sistema está persistindo os dados; **Quando** algum campo obrigatório estiver ausente; **Então** o registro não deve ser concluído. **CA-16-03**: **Dado** que ocorre uma falha durante o salvamento; **Quando** o erro é detectado; **Então** o sistema deve tentar persistir novamente automaticamente. **CA-16-04**: **Dado** que a falha persiste após tentativa de salvamento; **Quando** o sistema não consegue concluir a persistência; **Então** deve notificar o usuário sem perder os dados já coletados em memória. |
| **HU-17** | Melhor Resultado por Labirinto | Should Have | Como avaliador, eu quero ver rapidamente o melhor resultado registrado para um labirinto, para ter uma referência imediata de desempenho máximo. | **CA-17-01**: **Dado** que o avaliador acessa o detalhe de um labirinto; **Quando** a página carrega; **Então** um destaque visual deve apresentar a corrida com o menor tempo de conclusão (entre as marcadas como "Desafio Concluído"). **CA-17-02:** **Dado** que um novo resultado supera o melhor anterior; **Quando** a corrida é finalizada e salva; **Então** o destaque de melhor resultado deve ser atualizado automaticamente.  **CA-17-03**: **Dado** que nenhuma corrida foi concluída com sucesso no labirinto; **Quando** o usuário acessa o detalhe; **Então** o sistema web deve exibir "Nenhum desafio concluído ainda" no campo de destaque. |
| **HU-18** | Histórico do Labirinto | Must Have | Como avaliador, eu quero visualizar o histórico das execuções de um labirinto, para comparar resultados e identificar padrões de desempenho. | **CA-18-01:** **Dado** que o avaliador acessa a página de um labirinto específico; **Quando** a página carrega; **Então** deve ser exibida uma lista com todas as execuções registradas para aquele labirinto, mostrando: data/hora, tempo de conclusão, status do desafio e velocidade média.  **CA-18-02:** **Dado** que a lista de histórico é exibida; **Quando** o avaliador clica em uma corrida específica; **Então** o sistema web deve navegar para a visualização detalhada daquela execução. **CA-18-03:** **Dado** que múltiplas corridas estão registradas; **Quando** a lista é exibida; **Então** as corridas com "Desafio Cumprido" devem ter algum destaque visual que as diferencie das não cumpridas. |
| **HU-19** | Consulta Geral | Must Have | Como avaliador, eu quero visualizar e consultar todas as corridas registradas no sistema, para obter uma visão geral do desempenho do Micromouse. | **CA-19-01:** **Dado** que o avaliador acessa a tela de consulta; **Quando** a página é carregada; **Então** o sistema deve exibir todas as corridas registradas em ordem cronológica decrescente **CA-19-02:** **Dado** que existem corridas listadas; **Quando** o avaliador acessa uma corrida;  **Então** cada registro deve apresentar data/hora, tipo de labirinto, tempo de conclusão, status do desafio e velocidade média **CA-19-03**: **Dado** que o avaliador deseja visualizar corridas de um tipo específico de labirinto; **Quando** seleciona um filtro (4x4, 8x8 ou 16x16); **Então** o sistema deve atualizar a lista exibindo apenas as corridas correspondentes sem recarregar a página **CA-19-04:** **Dado** que um filtro está aplicado; **Quando** não existem corridas para o tipo de labirinto selecionado; **Então** o sistema deve exibir a mensagem "Nenhuma corrida registrada para este labirinto" **CA-19-05**: **Dado** que não há corridas registradas no sistema; **Quando** o avaliador acessa a consulta; **Então** o sistema deve exibir a mensagem "Nenhuma corrida registrada” |
| **HU-21** | Consulta Individual | Should Have | Como Avaliador, quero visualizar todos os dados de uma corrida específica para analisar o desempenho do Micromouse nessa corrida. | **CA-21-01:** **Dado** que há corridas salvas; **Quando** o avaliador acessa a página de uma corrida; **Então** o sistema deve exibir todos os dados da corrida na tela. |


###  **Requisitos Não-Funcionais de Software**

| ID | Categoria | Requisito Não Funcional | Priorização (MoSCoW) | Rastreabilidade |
| ----- | ----- | ----- | ----- | ----- |
| **RNF-01- SOFT** | Confiabilidade | O sistema web web deve detectar a ausência de pacotes de telemetria e alterar o status de conexão para "Offline" em no máximo 1 segundo após a interrupção, conforme definido em CA-09-01.	 | Must Have | HU-09 |
| **RNF-02- SOFT** | Confiabilidade | Caso a conexão bluetooth seja perdida, o sistema web web deve retomar a recepção e processamento de dados automaticamente, sem necessidade de reiniciar a sessão de monitoramento ativa.	 | Must Have | HU-08 |
| **RNF-03- SOFT** | Eficiência de Desempenho | RNF-EP-01	A interface de monitoramento deve refletir novos dados de telemetria (mapa, bateria, velocidade, tempo) em no máximo 500 ms após o recebimento do pacote pelo backend.	 | Must Have | HU-12/HU-13 |
| **RNF-04- SOFT** | Eficiência de Desempenho | A filtragem por tipo de labirinto na tela de histórico deve ocorrer sem recarregamento de página e produzir resultado visível em no máximo 300 ms após a seleção do filtro.	 | Must Have | HU-19 |
| **RNF-05- SOFT** | Compatibilidade | O módulo de recepção de dados deve ser compatível com o protocolo de transmissão XXXXXX Bluetooth, processando pacotes no formato e frequência acordados.	 | Must Have | HU-08 |
| **RNF-06- SOFT** | Usabilidade | Os indicadores de status de conexão ("Online"/"Offline") devem ser distinguíveis por cor, ícone e texto simultaneamente. | Should Have | HU-09 |
| **RNF-07- SOFT** | Usabilidade | O mapa do labirinto deve diferenciar visualmente, por cor e/ou símbolo, a posição atual do robô, as células visitadas e a trajetória percorrida.	 | Must Have | HU-12 |
| **RNF-08- SOFT** | Usabilidade  | Alertas críticos (bateria, parada inesperada, queda de conexão) devem estar visíveis na área principal da tela de monitoramento sem necessidade de scroll, rolagem horizontal ou interação adicional do usuário.	 | Should Have | HU-15 |
| **RNF-09- SOFT** | Usabilidade | Mensagens de erros devem ser redigidas em linguagem orientada ao usuário, sem expor mensagens de erro técnicas ou códigos internos.	 | Must Have | HU-10/HU-16 |
| **RNF-10- SOFT** | Manutenibilidade | O sistema web deve ser organizado em módulos independentes com responsabilidades bem definidas: (1) comunicação Bluetooth, (2) validação de telemetria, (3) renderização da interface e (4) persistência de dados, cada módulo alterável sem impacto nos demais.	 | Must Have | HU-08/HU-10/HU-12/HU-16 |
| **RNF-11- SOFT** | Manutenibilidade | A cobertura de testes do módulo de validação de telemetria e da lógica de disparo de alertas críticos deve ser de no mínimo 80% de cobertura de linhas. | Must Have | HU-10/HU-15 |
